SeaweedFS S3 Object Store Architecture

The quick-reference companion to the architecture lessons. Keep this open while working through the lessons.

The Four Layers

SeaweedFS's S3 object store is built from four distinct layers. Each layer has a clear responsibility, and layers communicate through well-defined interfaces.

Layer 4: S3 API Server (port 8333) — REST → filer protocol
Translates S3 REST (PutObject, GetObject, CreateBucket, etc.) into filer directory/file operations. Handles SigV4 auth, IAM, bucket policies, CORS, versioning, SSE encryption, multipart uploads, lifecycle. Stateless — multiple instances can run behind a load balancer.
Layer 3: Filer — directory/file namespace → blob references
Provides a hierarchical directory/file view on top of the flat blob store. Directories and file metadata (name, size, chunks, extended attributes) are stored in a pluggable metadata store (MySQL, Postgres, Redis, etc.). File contents are stored as references (chunks) to blobs on volume servers. The filer is stateless — all persistent state lives in the metadata store and volume servers.
Layer 2: Volume Servers — blob storage
Store data in 32 GB append-only volume files. Each volume contains many blobs (needles) stored sequentially. A 16-byte in-memory index maps each blob's file key to its offset and size within the volume. Reads are O(1): one disk seek to the offset, one read of the size. Write an ID with metadata to your own DB, not SeaweedFS's. The filer handles this mapping for you when using the S3 API.
Layer 1: Master Server — cluster coordination
Tracks which volume servers hold which volumes (volume-id → server mapping). Assigns new file keys for write requests. Supports automatic failover with multiple masters using Raft. The master is NOT in the data path — clients cache volume locations and talk directly to volume servers for reads and writes.

How a Bucket is Stored

  1. When CreateBucket("photos") is called, the S3 API server calls mkdir("/buckets", "photos") on the filer.
  2. The filer creates a directory entry at /buckets/photos in its metadata store.
  3. Bucket configuration (owner, versioning, object lock, ACL, lifecycle rules) is stored as extended attributes on the filer directory entry.
  4. The S3 API server also creates a collection on the volume servers with a name derived from the bucket. This isolates the bucket's data from other buckets.
  5. Bucket metadata is cached in the BucketRegistry (an in-memory LRU cache) to avoid repeated filer lookups.

How an Object is Stored and Retrieved

PutObject

  1. S3 API receives PUT /photos/sunset.jpg with SigV4 credentials.
  2. The IAM layer authenticates the request and checks bucket policy.
  3. The S3 API constructs the filer path: /buckets/photos/sunset.jpg via toFilerPath("photos", "sunset.jpg").
  4. The object is written to the filer, which splits it into chunks (typically 4 MB each) and assigns each chunk a file ID on a volume server.
  5. The filer entry for sunset.jpg stores: file name, total size, content type, ETag, and a list of chunks (each with volume_id + file_key + offset + size).
  6. The S3 API returns the ETag and optional version-id to the client.

GetObject

  1. S3 API receives GET /photos/sunset.jpg.
  2. The filer entry is looked up: /buckets/photos/sunset.jpg → list of chunk references.
  3. Key optimization: instead of proxying through the filer, the S3 API server streams chunks directly from volume servers using the ReaderCache. This eliminates the filer as a data-path bottleneck.
  4. SSE decryption (if applicable) happens inline during streaming.

How Users and IAM Work

  1. Credentials (access key + secret key pairs) are stored in a pluggable CredentialManager. Default: stored on the filer under /etc/iam/identities/.
  2. Authentication: Every S3 request is authenticated via AWS SigV4 signature verification. Anonymous access is possible only when IAM is disabled.
  3. Authorization: After auth, the IAM layer checks whether the identity has permission for the requested action on the target resource. This can use:
    • Simple per-identity actions (read, write, admin, list, tagging)
    • Bucket policies (JSON-based, evaluated by the PolicyEngine)
    • Advanced IAM with roles, OIDC federation, and STS temporary credentials
  4. Bucket Ownership: When a bucket is created, the creator's identity ID is stored as an extended attribute on the bucket's filer entry. Only the owner (or an admin) can delete the bucket.

Key Design Principles

PrincipleHow SeaweedFS Achieves It
No central metadata bottleneck Object metadata (file names, sizes, chunk lists) lives in the filer's metadata store, which is a horizontally-scalable external database. The master only tracks volumes, not objects.
O(1) data access Blob metadata (16 bytes per blob) fits entirely in memory on volume servers. One lookup + one disk seek + one read per blob.
Separation of metadata and data paths Metadata flows through the filer (what chunks exist, where they are). Data flows directly from volume servers to clients, bypassing the filer entirely.
Stateless API layer The S3 API server holds no persistent state. It can be scaled horizontally. All state is in the filer's metadata store and on volume servers.
Append-only storage Volumes are append-only logs. No in-place updates. Deletions are soft — they write a tombstone. Compaction reclaims space asynchronously. This is SSD-friendly and crash-safe.
