Glossary

A quick-reference glossary of terms used throughout the SeaweedFS architecture lessons. New terms are added as they appear.

Core Architecture

Master Server
The central coordinator. Stores volume-id → volume-server mappings (static, cacheable metadata). Generates monotonically-increasing file keys for write requests. Supports automatic failover with multiple masters — no single point of failure.
Volume Server
The data plane. Manages multiple volumes on disk. Stores blob metadata (16 bytes per blob) entirely in memory for O(1) lookups. Serves read/write requests directly from clients — the master is not in the data path.
Volume
A 32 GB (by default) append-only log file on a volume server. Contains many blobs stored sequentially. Volumes are the unit of replication, tiering, and garbage collection. Each has a unique 32-bit volume ID.
Needle / Blob
The atomic unit of storage. A needle is a key-value pair: a 64-bit file key, a 32-bit file cookie (for security), and the blob contents. Stored sequentially inside a volume.
FID (File ID)
A triple of (volume_id, file_key, file_cookie) that uniquely addresses any blob in the cluster. Rendered as 3,01637037d6 in the HTTP API.
Filer
A stateless, linearly-scalable server that adds a directory/file namespace on top of the flat blob store. Stores directory structure in a pluggable metadata store (MySQL, Postgres, Redis, LevelDB, etc.). File contents point to one or more chunks (blobs) on volume servers. Exposes HTTP, WebDAV, FUSE, and S3 interfaces.

S3 API Concepts

S3 API Server
A stateless HTTP server that translates S3 REST calls (PutObject, GetObject, ListBuckets, etc.) into filer operations. Runs alongside or separately from the filer. Supports path-style and virtual-host-style bucket addressing.
BucketsPath
The filer directory prefix where all S3 buckets live. Default is /buckets. A bucket named my-bucket lives at /buckets/my-bucket on the filer. Objects within the bucket live as files under that directory.
bucketDir
The absolute filer path for a bucket: bucketsPath/bucketName. Defined in weed/s3api/bucket_paths.go. Used to construct object paths.
toFilerPath
The function that maps an S3 (bucket, object-key) pair to a filer path: bucketsPath/bucket/normalizedObjectKey. The object key is normalized first (e.g., removing redundant slashes).
BucketRegistry
An in-memory LRU cache (weed/s3api/bucket_metadata.go) that lazily loads bucket metadata from the filer. Uses singleflight to deduplicate concurrent loads. Avoids hitting the filer for every request.
BucketMetaData
The in-memory representation of a bucket: name, owner, ACL grants, object ownership setting, and whether it's a table bucket.
Collection
A storage namespace in the volume layer. Each bucket gets its own collection (mapping to a prefix on volume servers) so that data from different buckets can be placed on different volumes or tiers.

Identity and Access

IdentityAccessManagement (IAM)
The subsystem (weed/s3api/) that handles AWS SigV4 authentication, credential verification, and action authorization. Supports credentials stored in filer, Postgres, or memory.
CredentialManager
Pluggable backend for storing S3 identities and their secret keys. Supports filer-based storage (filer_etc, the default), PostgreSQL, and in-memory.
IAMManager
The advanced IAM system (weed/iam/integration/) that adds OIDC federation, STS (temporary credentials), roles, and fine-grained policies. Configured via a JSON config file.
STS
Security Token Service. The S3 API server can issue temporary credentials via AssumeRoleWithWebIdentity (OIDC/JWT), AssumeRole (SigV4), and GetCallerIdentity.

Data Flow

Chunk
A reference to a blob stored on a volume server, described by a file ID and byte range. Large S3 objects are split into multiple chunks by the filer. Each chunk points to a volume server + fid + offset + size.
ReaderCache
A shared, server-lifetime cache (weed/filer/) for in-flight and recently-completed chunk downloads. Multiple concurrent GET requests for the same object share a single download — the S3 API server uses this to stream directly from volume servers, bypassing the filer for data transfer.
Stream-from-volume-servers
The optimization where the S3 API server reads chunks directly from volume servers instead of proxying through the filer. Eliminates ~19ms of filer proxy overhead on GET requests.
