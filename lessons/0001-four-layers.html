Lesson 1: The Four Layers of the SeaweedFS Object Store

Before we can understand how SeaweedFS distributes particular kinds of data (S3 objects, buckets, users), we need to understand the layers that all data flows through. This lesson introduces the four-layer architecture and how each layer contributes to the distribution of data and metadata.

The Problem: Why Layers?

In a traditional distributed filesystem (like HDFS), a central NameNode stores a mapping of every filename to its chunks and chunk locations. This works for large files, but breaks down with billions of small files — the NameNode's in-memory metadata becomes the bottleneck.

SeaweedFS solves this by splitting the problem across four layers with different responsibilities. No single component holds all the metadata.

The Four Layers

Layer 1: Master Server — The Lightweight Coordinator

The master stores only volume-id → volume-server mappings and generates file keys for new writes. It does not know about files, directories, buckets, or users. Its data set is small and stable — typically a few thousand entries, even for petabyte-scale clusters.

This is the key insight from Facebook's Haystack design paper: by making volumes (not files) the unit of management, the coordinator stays tiny.

Layer 2: Volume Servers — The Data Plane

Volume servers store data in 32 GB append-only volume files. Each volume is a sequential log of blobs (also called needles). A 16-byte in-memory index maps each blob's file key → (offset, size) within the volume. One disk read per blob access.

Volume servers are entirely self-describing: they know everything about the blobs they store and need no master help to serve reads. They also handle replication (rack/data-center aware), compression, and TTL expiration.

Layer 3: Filer — The Namespace Layer

The filer adds a hierarchical directory/file view on top of the flat blob store. It stores directory structure and file metadata (name, size, content type, chunk list) in a pluggable external database — MySQL, Postgres, Redis, LevelDB, etc.

A file in the filer doesn't contain data; it contains chunk references. Each chunk is a pointer to a blob on a volume server: (volume_id, file_key, offset, size). Large files are split across multiple chunks, potentially on different volume servers.

Critical insight: The filer stores metadata, not data. When you GET an object, the filer tells the S3 API where the chunks are, but the actual bytes are streamed directly from volume servers to the client. The filer is never a data-path bottleneck.

Layer 4: S3 API Server — The Translation Layer

The S3 API server is a stateless HTTP server that translates S3 REST calls into filer operations:

  • CreateBucket("photos") → mkdir("/buckets/photos") on the filer
  • PutObject("photos", "sunset.jpg") → write file at /buckets/photos/sunset.jpg via the filer
  • GetObject("photos", "sunset.jpg") → read filer entry for /buckets/photos/sunset.jpg, stream chunks directly from volume servers
  • ListBuckets() → list directories under /buckets

It also handles authentication (AWS SigV4), authorization (IAM policies), CORS, versioning, SSE encryption, multipart uploads, and lifecycle policies. All of this is implemented in weed/s3api/.

How Data and Metadata are Distributed

WhatWhere It LivesHow It Scales
Object data (bytes) Volume servers, as blobs in append-only volume files Add more volume servers. Volumes are the unit of distribution.
Object metadata (name, size, chunks, ETag) Filer metadata store (Postgres, Redis, etc.) Scale the filer metadata store independently (e.g., sharded Postgres).
Bucket metadata (owner, versioning, ACL, lifecycle, object lock) Extended attributes on the filer directory entry for the bucket (e.g., /buckets/photos) Same path as object metadata — scales with the filer store. Cached in-memory by BucketRegistry.
User credentials (access keys, secrets) Filer (/etc/iam/identities/) or Postgres or memory Scales with the chosen credential store.
IAM policies (bucket policies, roles) Filer (/etc/iam/policies/) or in-memory Loaded at startup or from config; evaluated in-memory by each S3 API server.
Volume-to-server mapping Master server (in memory, replicated via Raft) Thousands of volumes per master; millions handled by the volume-id → server lookup. Stable and cacheable.

Trace It: PutObject and GetObject

Here is the full lifecycle of an S3 object, traced through every layer:

PutObject("photos", "sunset.jpg")

1. S3 API: Authenticate request (SigV4). Check IAM policy. Normalize object key.
2. S3 API → Filer: Call toFilerPath("photos", "sunset.jpg") → /buckets/photos/sunset.jpg. Write the data to this filer path.
3. Filer: Split the data into 4 MB chunks. For each chunk, request a new file ID from the master.
4. Master: Pick a writable volume server. Return (volume_id, file_key, file_cookie, volume_server_url).
5. Filer → Volume Server: Write each chunk's blob to its assigned volume server.
6. Filer: Store the file entry in its metadata DB: name="sunset.jpg", size=..., chunks=[{vid, fid, offset, size}, ...], extended attributes (ETag, content-type, SSE info, etc.).
7. S3 API: Return ETag and (if versioned) version-id to client.

GetObject("photos", "sunset.jpg")

1. S3 API: Authenticate. Look up filer entry at /buckets/photos/sunset.jpg — get the chunk list.
2. S3 API → Volume Servers (directly!): For each chunk reference, stream the blob bytes directly from the volume server. No filer in the data path.
3. S3 API: Assemble the response, handle Range requests, apply SSE decryption, set response headers (ETag, Content-Type, etc.).
