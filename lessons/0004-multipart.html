Lesson 4: Multipart Upload Lifecycle

S3 multipart upload uploads large objects in parts, assembled at the end into a single object. This lesson traces each phase through the code.

The Five Phases

1. Initiate — CreateMultipartUpload
Creates a unique upload ID bound to the object key. Creates a staging directory /buckets/<b>/.uploads/<id>/. No data uploaded yet.
2. Upload — UploadPart (once per part)
Each part uploaded independently using the chunked pipeline from Lesson 3. Parts can arrive in any order.
3. Complete — CompleteMultipartUpload
Validates ETags, concatenates chunk references with new offsets, writes final object, cleans up staging directory.
4. Abort (optional) — AbortMultipartUpload
Deletes the staging directory and all parts.
5. Lifecycle (background)
Expired uploads are auto-aborted by the lifecycle subsystem.

Phase 1: Initiate

createMultipartUpload (filer_multipart.go:61) generates the upload ID:

uploadId := SHA1(objectKey) + "_" + UUID

The SHA-1 prefix cryptographically binds the upload to the object key. checkUploadId verifies this at every subsequent call — you cannot upload parts for one key and complete as another.

A staging directory is created at /buckets/<bucket>/.uploads/<uploadID>/. Its extended attributes store: the object key, user metadata, content type, SSE encryption config, checksum algorithm, object lock metadata.

Phase 2: UploadPart

PutObjectPartHandler (s3api_object_handlers_multipart.go:330) handles each part:

filePath = "/buckets/my-bucket/.uploads/SHA_UUID/0003_UUID.part"

The filename format is NNNN_UUID.part — zero-padded part number for sorted listing + random UUID so re-uploading the same part number creates a new file.

Upload uses the exact same putToFiler pipeline as PutObject (8 MB chunks, parallel upload, direct to volume servers). Key differences from PutObject:

  • Parts are independent — part 7 can upload before part 1.
  • Re-uploads: same part number → new .part file. At completion, if duplicates exist, newest wins (by chunk ModifiedTsNs).
  • SSE encryption inherited from the upload directory entry — client doesn't re-send SSE headers per part.
  • Lifecycle TTL suppressed — parts get lifecycleTTLSec=0. The clock starts at completion, not upload.
  • 5 MB minimum per part (except last), validated at completion.

Phase 3: CompleteMultipartUpload

completeMultipartUpload (filer_multipart.go:574) has five steps:

Step 1: Validate the part list

The client submits {PartNumber, ETag} pairs. The server checks parts are in ascending order with valid ETags. Re-ordered parts → ErrInvalidPartOrder.

Step 2: Match parts to stored files

Lists the staging directory, finds all .part files. For each part number: validates ETag, picks newest entry if duplicates exist, verifies size ≥ 5 MB (except last). Unmatched entries go to the delete list.

Step 3: Assemble chunks

No data is copied. Chunk references from each part are concatenated with recomputed offsets:

offset = 0
for part in ordered_parts:
    for chunk in part.chunks:
        finalChunk = { ...chunk..., Offset: offset }
        offset += chunk.Size

Step 4: Write the final object

Versioning stateBehavior
EnabledVersion file in .versions/ with real version ID. Latest-version pointer updated.
SuspendedObject written at path with versionId="null". Existing null-version replaced.
NoneObject written directly at path.

Extended attributes inherit from the upload directory (metadata, content type, object lock, checksum). SSE metadata is copied from the first part. The multipart ETag is stored.

Step 5: Clean up

Stale/duplicate part entries are deleted. The upload directory is removed. Completion is idempotent — if the object already exists with the same upload ID, the call returns the existing result (handles network retries).

The Multipart ETag

Unlike PutObject's content-MD5 ETag, the multipart ETag is the MD5 of concatenated per-part ETags + part count:

ETag = MD5(etag1_raw || etag2_raw || ... || etagN_raw) + "-" + str(N)

For example: "d41d8cd98f00b204e9800998ecf8427e-3". This matches AWS S3 behavior.

Phase 4: Abort

abortMultipartUpload (filer_multipart.go:937) recursively deletes the upload directory. All .part files are gone. The volume server blobs become orphans, eventually reclaimed by volume compaction.
