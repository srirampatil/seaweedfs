Lesson 6: Lifecycle Policy Processing

S3 lifecycle policies automatically expire objects, delete old versions, and abort abandoned multipart uploads. SeaweedFS implements this as a two-pronged system: an event-driven meta-log replay for continuous evaluation, and a periodic bucket walker for bootstrap and rules that can't be driven by events alone.

Lifecycle Rule Evaluation

Rules are stored as XML on the bucket (PutBucketLifecycleConfiguration) and parsed into a flat Rule struct (s3lifecycle/rule.go:7):

type Rule struct {
    ID     string
    Status string   // "Enabled" | "Disabled"
    Prefix string
    FilterTags map[string]string
    FilterSizeGreaterThan int64
    FilterSizeLessThan    int64

    ExpirationDays            int
    ExpirationDate            time.Time
    ExpiredObjectDeleteMarker bool
    NoncurrentVersionExpirationDays int
    NewerNoncurrentVersions         int
    AbortMPUDaysAfterInitiation     int
}

EvaluateAction (s3lifecycle/evaluate.go:12) checks whether a rule fires for a given object at a given time. Each action kind has different gating:

Action kindGatesAction
ExpirationDaysIsLatest, is not delete marker, ModTime + Days ≤ nowDelete object (respecting versioning state)
ExpirationDateIsLatest, is not delete marker, now ≥ ExpirationDateDelete object
NoncurrentDaysNOT IsLatest, SuccessorModTime + Days ≤ now, optionally gated by NewerNoncurrentDelete specific version
NewerNoncurrentNOT IsLatest, NoncurrentIndex ≥ count, no NoncurrentDaysDelete specific version
ExpiredDeleteMarkerIsLatest, is delete marker, NumVersions == 1Remove the delete marker
AbortMPUIsMPUInit=true, ModTime + Days ≤ nowAbort multipart upload
NoncurrentDays relies on SuccessorModTime, not the version's own ModTime. When a new version replaces this one, markVersionNoncurrent stamps ExtNoncurrentSinceNs on the demoted entry. If that timestamp exists, it's used as SuccessorModTime; otherwise the engine falls back to the version's own mtime (older code paths).

Architecture: Five Components

1. Engine — Rule Compilation
Compiles enabled rules into an in-memory Snapshot. Pre-builds delay-grouped action indexes so matching is O(1) per event. Computes content hashes to detect rule changes. Splits rules into replay-eligible (event-driven) and walker-only partitions.
2. Reader — Meta-Log Subscription
Subscribes to the filer meta-log via gRPC. One shared subscription for all shards. Emits Event structs (TsNs, Bucket, Key, OldEntry, NewEntry) to a fan-out channel. Events carry pre-computed sibling state for versioned objects.
3. Router — Match Events to Rules
Receives events from the reader. For each event, looks up matching rules from the engine snapshot. Evaluates each (rule, kind, event) tuple via EvaluateAction. Produces Match tuples with DueTime, ObjectKey, VersionID, Identity for CAS.
4. Daily Run — Orchestration
Orchestrates per-shard replay and periodic walking. One goroutine per shard. Each shard: runs recovery/cold-start walker if needed → runs steady-state walker (throttled) → drains events from its channel → routes matches → dispatches via LifecycleDelete RPC → persists cursor.
5. S3 API Server — LifecycleDelete Handler
Receives LifecycleDelete RPC: re-fetches the live entry → verifies identity CAS (MtimeNs, Size, HeadFid, ExtendedHash) → checks object lock → dispatches by action kind to actual S3 operations.

Event-Driven Replay: The Meta-Log Path

This is the primary evaluation path for time-based rules (ExpirationDays, NoncurrentDays, AbortMPU). It works by subscribing to the filer's meta-log:

  1. Shared subscription: One gRPC stream subscribes to all shards. A fan-out goroutine routes events to per-shard channels by ev.ShardID.
  2. Shard draining: Each shard goroutine drains events from its channel up to runNow (the pass boundary). Events past runNow cancel the reader — they will be picked up on the next pass.
  3. Due-time calculation: The router skips events whose ModTime + rule TTL hasn't yet exceeded runNow. Events whose due time lies in the future stay in the meta-log for future passes.
  4. Cursor advancement: After draining, the per-shard cursor is persisted with TsNs (last processed event timestamp), RuleSetHash, and PromotedHash. Next pass resumes from here.

The Walker: Bootstrap and Replay-Gap Coverage

Not all actions can be driven by meta-log events alone. The walker (dailyrun/walk_buckets.go) does a full bucket subtree scan to cover:

  • ExpirationDate rules — a fixed calendar date. The object's ModTime is irrelevant; what matters is "is now past 2026-12-31?" This can't be expressed as a meta-log TTL.
  • ExpiredDeleteMarker rules — requires knowing NumVersions, which the meta-log event alone doesn't carry.
  • NewerNoncurrent (count-based) — requires knowing NoncurrentIndex, which needs sibling listing.
  • Cold start / rule change — objects PUT before the rule existed have no meta-log event within the replay window. The walker catches them.

The walker runs on its own throttled cadence (WalkerInterval). It's not triggered every pass — typically hours between walks for production clusters. The walker dispatches via the same LifecycleDelete RPC, but without identity CAS (the server trusts bootstrap calls from a fresh scan).

Sharding and Parallelism

Rules and objects are sharded across ShardCount partitions (default 16). Each shard has:

  • Its own cursor (TsNs — position in the meta-log stream)
  • Its own goroutine in the daily run
  • Its own event channel (buffer 256)

This means 16 shards can replay the meta-log in parallel, with dispatch rate-limited by a shared rate.Limiter. The fan-out from the single subscription to per-shard channels is lock-free via select.

Identity CAS: Preventing Stale Dispatch

Between rule evaluation (router) and dispatch (server), the object might have been modified. To prevent dispatching on the wrong version:

  1. The router captures an EntryIdentity at match time: {MtimeNs, Size, HeadFid, ExtendedHash}.
  2. This identity is serialized into the LifecycleDelete RPC.
  3. On the server, LifecycleDelete re-fetches the live entry and compares identities.
  4. If identities don't match → NOOP_RESOLVED: STALE_IDENTITY. The object has moved on; the old deletion verdict is discarded.

This is a compare-and-swap pattern: the server only acts if the entry at dispatch time matches the entry at evaluation time.

Server-Side Dispatch

LifecycleDelete (s3api_internal_lifecycle.go:22) handles the actual deletion:

ActionKindVersioning enabledVersioning suspendedNo versioning
ExpirationCreate delete markerDelete null + create delete markerDelete object
NoncurrentDelete specific version file .versions/v_<id>
ExpiredDeleteMarkerDelete the marker version file
AbortMPUDelete .uploads/<id>/ directory

Before acting, the server also checks object lock protections — lifecycle never bypasses governance or compliance retention.

Fast-Path: Volume TTL

For simple Expiration.Days rules on non-versioned buckets, SeaweedFS has an optional fast path: the lifecycle TTL is stamped directly onto volume server blobs at write time. When the TTL expires, the volume server reclaims the space automatically — no lifecycle worker involvement needed. This is opt-in per bucket (ExtLifecycleTtlFastPathKey).
