Lesson 7: The Filer Meta-Log

The meta-log is the backbone of SeaweedFS's event-driven architecture. Every change to the filer namespace — create, update, delete, rename — is recorded as a log entry. Subscribers consume this log to react to changes: the S3 API server refreshes bucket configs and IAM credentials; the lifecycle worker evaluates expiration rules; filer sync replicates across clusters.

What the Meta-Log Is (and Isn't)

The meta-log is not a separate storage subsystem. It has no WAL, no dedicated disk partition, no special ring buffer. It is simply regular files on the filer itself, written to a reserved directory tree:

/topics/.system/log/
  2026-07-19/
    15-13.00000001    ← YYYY-MM-DD/HH-MM.filerId
    15-14.00000001

Each log file is a normal filer file — its content is chunked across volume servers like any other file. The only special thing about it is the path prefix.

No reserved storage. The log grows unboundedly as append-only files. There is no ring buffer, no retention window, no automatic truncation. Old day directories persist until something explicitly deletes them.

The LogBuffer: In-Memory First, Disk Later

Events don't go directly to disk. They accumulate in an in-memory LogBuffer (util/log_buffer/log_buffer.go:62):

PropertyValue
Buffer size8 MB
Sealed buffers retained4 previous buffers
Time-based flushEvery 1 minute (LogFlushInterval)
Size-based flushWhen 8 MB buffer fills before the minute ends
Flush queue depth16 entries

The buffer has two goroutines: loopFlush drains the flush queue and calls logFlushFunc, and loopInterval fires a time-based flush every minute. When the buffer fills mid-minute, the write path itself triggers an immediate flush.

Four previous sealed buffers are kept in the prevBuffers ring. This serves newly-arriving subscribers who started reading from a timestamp that falls within one of those sealed windows — they get data from memory without needing to read persisted log files from disk.

Wire Format

Each entry in the buffer (and on disk) is:

[4-byte big-endian uint32 size][protobuf-marshaled filer_pb.LogEntry]

The LogEntry carries:

message LogEntry {
    int64 ts_ns              = 1;   // event timestamp (nanoseconds)
    int32 partition_key_hash = 2;   // hash of directory path (for sharding)
    bytes data               = 3;   // marshaled SubscribeMetadataResponse
    bytes key                = 4;   // the directory (partition key)
    int64 offset             = 5;   // sequential offset within this LogBuffer
}

The data field is a marshaled SubscribeMetadataResponse containing the actual event: old entry (nil for creates), new entry (nil for deletes), whether to delete chunks, the parent path for renames, and signatures from all filers that have applied this event.

Event Generation: Unconditional

Every mutation to the filer — every create, update, delete, and rename across the entire namespace — calls NotifyUpdateEvent → logMetaEvent → AddDataToBuffer. There is no gate on lifecycle policies. The meta-log is generated regardless of whether anyone is subscribed.

Two things are excluded:

  1. SystemLogDir entries (filer_notify.go:44): mutations under /topics/.system/log/ don't generate events. Without this guard, writing a log file would generate a log event, which would get logged, generating another event — infinite recursion.
  2. metadataEventsSuppressed: callers can suppress event generation via context. The MetaAggregator uses this when replaying peer events into the local store — those events already exist in the peer's log.

Durability Model: CDC, Not WAL

The call order in CreateEntry (filer.go:259,278) tells the story:

f.Store.InsertEntry(ctx, entry)    // ← metadata store write FIRST
f.NotifyUpdateEvent(ctx, ...)       // ← log event SECOND

The metadata store mutation happens before the log event. If the filer crashes between these two lines, the entry is safely persisted in Postgres/Redis/LevelDB, but no log event was ever emitted.

CDC, not WAL. The meta-log is a change-data-capture stream built on top of the metadata store. The source of truth is the store itself. The meta-log provides at-most-once delivery with a crash-loss window of up to 1 minute (flush interval) or 8 MB (buffer size).

Why Lost Events Are Acceptable

Each subscriber has its own recovery mechanism that doesn't depend on a complete log:

SubscriberRecoveryCost
Lifecycle worker Cold-start walker scans the full namespace; steady-state walker runs periodically (hours) Walker throttled by WalkerInterval — not per-missed-event
S3 API config/IAM LoadS3ApiConfigurationFromCredentialManager() reads complete state at startup A single full reload, then live events keep it fresh
Filer sync TraverseBfsMetadata does a full tree traversal on first peer connection One-time, then incremental via the meta-log stream

CDC vs. Snapshot Diff: Correctness vs. Engineering

A natural question: why capture every individual change? Wouldn't periodic snapshots with diffs be more efficient?

Upon closer analysis, the question is more nuanced than it first appears. Snapshot diffs can produce correct lifecycle outcomes because version files are self-describing — they carry their own timestamps. The differences are primarily engineering tradeoffs.

Why snapshot diffs CAN work for lifecycle

Each version file in .versions/ carries its own metadata, written at mutation time and persisted on the entry:

  • ExtNoncurrentSinceNsKey — exact nanosecond the version was demoted
  • ExtVersionIdKey — the version ID
  • ExtDeleteMarkerKey — whether this is a delete marker
  • Attributes.Mtime, Attributes.FileSize, ExtETagKey

For MPU init records in .uploads/, the creation timestamp is on the directory entry itself. A comprehensive snapshot that lists all entries under .versions/ and .uploads/ can read these attributes directly — no historical reconstruction needed.

Consider a version created and made noncurrent between snapshots:

T0 snapshot:        .versions/ has v_V1 (latest)
T+1h: V2 replaces V1 — V1.NoncurrentSinceNs = T+1h
T+2h: V3 replaces V2 — V2.NoncurrentSinceNs = T+2h
T+4h snapshot:      .versions/ has v_V1, v_V2, v_V3
                     V3 latest; V1/Noncurrent=T+1h, V2/Noncurrent=T+2h

Snapshot diff at T+4h lists all three versions. Each carries its own NoncurrentSinceNs. EvaluateAction computes due-time from the exact timestamp on the entry. Verdict: correct. Every surviving version is visible and evaluable.

The same holds for MPU abort: the init record's creation timestamp is on the .uploads// directory entry. The snapshot diff reads it directly.

What snapshot diff misses

The only versions invisible to a snapshot diff are those explicitly deleted by the user within the snapshot window:

T0 snapshot:        V1 (latest), V0 (noncurrent)
T+1h: Delete V0 — user calls DeleteObject?versionId=V0
T+4h snapshot:      V1 (latest). V0 is gone.

But this is not a failure. The user explicitly deleted V0 — lifecycle doesn't need to expire it. The lifecycle engine garbage-collects versions the user left behind. A user-initiated delete supersedes lifecycle and removes the blob data correctly.

The real tradeoffs

CDC (event log)Snapshot diff
Lifecycle outcomeCorrectCorrect (same outcome)
Discovery latencySub-second (event at mutation time)Up to T+N (snapshot interval)
Scan costOnly mutated entries matching rule prefixesAll entries in covered scope
NoncurrentDays precisionNanosecond (from entry attribute)Nanosecond (from entry attribute)
InfrastructureExists and runs todayWould need snapshot infrastructure
Intermediate statePreserved in event streamUser-deleted states invisible (acceptable)

The critical insight: each version file is self-describing. The diff doesn't need to reconstruct history — it reads the attributes written at mutation time and persisted on the entry. Snapshot diffs and CDC converge to the same lifecycle outcome; they differ only in when the evaluation happens (latency) and how much is scanned (cost).

Replay Efficiency

Snapshot diffs would eliminate the cursor-replay loop entirely. Instead of re-evaluating the same event 30 times for a 30-day rule, the diff evaluates each entry once. But the meta-log already optimizes this with cursor management:

if m.DueTime.After(runNow) {
    skippedAny = true      // future due-time → skip
    continue
}
if !stuck {
    cursorAdvanceTo = ev.TsNs  // advance cursor only if no skipped matches
}

The cursor only gets stuck on events that match a rule's prefix AND have a future due-time. Events outside all rule prefixes stream through freely. The ~30× re-evaluation overhead for pending-due objects is the price of event-driven evaluation; a snapshot diff would pay the price of full-scope scanning instead.

Mode 1: Server-Streamed (gRPC Push)

The SubscribeMetadata RPC (filer.proto:75) is a server-side streaming call. The client sends {since_ns, path_prefix, client_id} and the server:

  1. Catches up from disk: reads persisted log files from since_ns up to the first byte in the in-memory buffer, streaming each event.
  2. Switches to live: subscribes to the LogBuffer's subscriber notification channel. New events arrive with sub-millisecond latency.
  3. Batches backlog: during catch-up, multiple events can be packed into one response via SubscribeMetadataResponse.events.

Mode 2: Client-Direct Volume Server Reads

When the client sets client_supports_metadata_chunks: true, the server sends LogFileChunkRef messages instead of streaming individual events. Each ref contains the file ID chunks for a log file. The client then:

  1. Reads all chunk data directly from volume servers in parallel.
  2. Decodes [4-byte size | LogEntry] records locally.

This is the same optimization as the S3 GET path: the filer provides chunk references, and the client fetches data directly — the filer is never a byte-throughput bottleneck for historical log replay.

Two LogBuffers: Local vs. Aggregate

Each filer has two LogBuffers (filer.go:52,62):

BufferNameFlush to disk?Purpose
LocalMetaLogBuffer"local"Yes — logFlushFunc → appendToFileRecords this filer's own mutations
MetaAggregator.MetaLogBuffer"aggr"No — flushFn is nilMerges events from all peer filers for subscribers

The local buffer persists to disk. The aggregate buffer is memory-only — it serves live subscribers with a merged view of all filers' events. When a subscriber needs historical data, it falls back to reading persisted log files (which come from individual filers' local buffers).

Multi-Filer: Peers and Merge

In a multi-filer deployment, the MetaAggregator subscribes to every peer filer via SubscribeLocalMetadata:

  1. On first connection to a peer, it does a full metadata tree traversal (TraverseBfsMetadata) to import all pre-existing entries.
  2. Then it subscribes to the peer's live meta-log stream.
  3. Each received event is added to the aggregate MetaLogBuffer (memory only) and replayed into the local store (Replay function).
  4. The peer offset (TsNs) is periodically persisted into the filer's KV store so the subscription can resume from the right point after restart.

When reading persisted logs, an OrderedLogVisitor uses a min-heap priority queue across per-filer log file iterators, merging events in timestamp order.
