Lesson 3: The Data Path — Chunks, Blobs, and Direct Streaming

Lessons 1 and 2 covered the metadata — how bucket configs and IAM flow through the system, kept consistent by push-based subscriptions. This lesson covers the data: what actually happens to the bytes when you PutObject and GetObject.

The Core Data Structures

FileChunk: A Pointer to a Blob

A FileChunk (filer.pb.go:894) is SeaweedFS's fundamental data reference. It's not the data itself — it's a pointer that says "the bytes at offset N through offset N+M in this file live at this volume server, at this file ID, at this byte range":

type FileChunk struct {
    Fid         *FileId     // (volume_id, file_key, cookie) — where the blob lives
    Offset      int64       // byte offset within the logical file
    Size        uint64      // size of this chunk's contribution
    SseType     SSEType     // encryption type (NONE, SSE_C, SSE_KMS, SSE_S3)
    SseMetadata []byte      // per-chunk encryption metadata
    IsChunkManifest bool    // true if this chunk's content is itself a list of FileChunks
    CipherKey   []byte      // optional server-side encryption key
}

type FileId struct {
    VolumeId uint32   // which 32GB volume
    FileKey  uint64   // unique key within that volume
    Cookie   uint32   // security cookie (prevents URL guessing)
}
Key insight: Offset is the position within the logical file, not within the volume. A 100 MB object split into 25 chunks of 4 MB each would have chunk offsets 0, 4MB, 8MB, etc. — regardless of where each chunk lands on its volume server.

Filer Entry: The Full Object Metadata

A filer Entry for an S3 object contains everything needed to reconstruct the file:

  • Chunks ([]*FileChunk) — the list of blob pointers
  • Attributes (FuseAttributes) — size, MIME type, MD5, timestamps, TTL
  • Extended (map[string][]byte) — ETag, version ID, SSE metadata, user metadata, tags, storage class, object lock info
  • Content ([]byte) — for tiny files (≤ 256 KB), the data is stored inline in the entry itself

PutObject: From HTTP Bytes to Volume Blobs

Here is the complete PutObject data path, traced through putToFiler (s3api_object_handlers_put.go:373):

HTTP Body → SSE Encrypt → 8 MB Chunker → Parallel Upload → Volume Servers → Filer Entry

Step 1: SSE Encryption (If Requested)

Before chunking, the data stream is wrapped in an encryption layer (SSE-C, SSE-KMS, or SSE-S3). The encrypted stream is what gets chunked and uploaded — the volume servers never see plaintext.

Step 2: Auto-Chunking

operation.UploadReaderInChunks (operation/upload_chunked.go:59) reads from the encrypted stream in 8 MB chunks:

const chunkSize = 8 * 1024 * 1024  // 8 MB
const smallFileLimit = 256 * 1024  // 256 KB stored inline

Chunks are processed in parallel — up to 4 concurrent uploads. Each chunk independently goes through the assign→upload pipeline.

Step 3: Volume Assignment

For each chunk, the assignFunc calls filer.AssignVolume via gRPC, which delegates to the master server. The master:

  1. Picks a volume server with a writable volume in the correct collection (derived from the bucket name).
  2. Generates a new (volume_id, file_key, cookie) — the file ID.
  3. Returns the file ID and the volume server's URL.

This is the only master involvement in the entire data path. Once assigned, the chunk upload is direct to the volume server.

Step 4: Parallel Upload to Volume Servers

Each chunk is uploaded directly to its assigned volume server via HTTP. The volume server:

  1. Appends the blob to its append-only volume file.
  2. Records the (file_key → offset, size) mapping in its in-memory index.
  3. Returns success.

The UploadReaderInChunks result includes a list of FileChunk entries — one per uploaded segment — with each chunk's file ID, offset, and size.

Step 5: Create the Filer Entry

Once all chunks are uploaded, the S3 API server assembles a filer_pb.Entry and calls CreateEntry on the filer:

entry := &filer_pb.Entry{
    Name:        "sunset.jpg",
    IsDirectory: false,
    Attributes: &filer_pb.FuseAttributes{
        FileSize: chunkResult.TotalSize,
        Mime:     "image/jpeg",
        Md5:      md5Sum,
        Crtime:   now.Unix(),
    },
    Chunks:   chunkResult.FileChunks,  // [FileChunk, FileChunk, ...]
    Extended: map[string][]byte{
        "etag":         []byte(etag),
        "x-amz-meta-*": ...,           // user metadata
    },
}

The filer stores this entry in its metadata DB (Postgres, Redis, LevelDB, etc.). The actual blob data never passes through the filer — it went directly from the S3 API server to the volume servers.

One exception: files smaller than 256 KB bypass the chunking pipeline entirely. Their content is stored inline in entry.Content, and the filer entry itself carries the bytes. This avoids volume-server round-trips for tiny objects.

GetObject: Streaming Directly from Volume Servers

The read path is where SeaweedFS's architecture really shines. streamFromVolumeServers (s3api_object_handlers.go:935) eliminates the filer from the data path entirely:

Filer (entry lookup) → Chunk Resolution → ReaderCache → Volume Servers (direct) → SSE Decrypt → HTTP Response

Step 1: Look Up the Filer Entry

The S3 API server fetches the object's filer entry — getting the list of FileChunk references. This is a metadata-only operation: a single gRPC call to the filer's metadata store. No data is transferred yet.

Step 2: Resolve Chunks to Visible Intervals

The chunk list might contain chunk manifests — chunks whose content is itself a list of sub-chunks (used for very large files). NonOverlappingVisibleIntervals recursively resolves these into a flat list of (offset, size, file-id) intervals covering the requested byte range.

The volume server URLs are resolved via the FilerClient's vidMap cache — a cached mapping of volume IDs to volume server URLs. No gRPC call to the filer per chunk.

Step 3: Stream Through the ReaderCache

This is the optimization that makes SeaweedFS fast for concurrent reads. The ReaderCache (filer.NewReaderCache, created at server startup in s3api_server.go:284) is a shared, server-lifetime cache for in-flight and recently-completed chunk downloads.

How it works:

  1. A ChunkReaderAt is created, backed by the shared ReaderCache.
  2. When one request starts downloading chunk N from volume server X, the ReaderCache registers an in-flight download.
  3. If a second concurrent request also needs chunk N, it shares the same in-flight download — no duplicate network call.
  4. Completed chunks are cached in memory (configurable size, default 256 concurrent download slots).

The stream flows directly: volume server → ReaderCache buffer → HTTP response writer. The filer is not in the data path at all. This eliminates ~19ms of filer proxy overhead compared to the old path where data flowed through the filer.

Step 4: SSE Decryption (If Encrypted)

For encrypted objects, decryption happens inline during streaming. The S3 API server:

  1. Extracts the encryption key from the filer entry's extended attributes.
  2. Creates a decryption wrapper around the chunk reader (AES-256-GCM for SSE-C, key unwrapping for SSE-KMS).
  3. Streams decrypted bytes to the HTTP response.

For range requests on encrypted objects, only the needed chunks are fetched and decrypted — not the entire object.

Chunk Manifests: Scaling to Very Large Files

A single object can span thousands of chunks (e.g., a 100 GB object = 12,500 chunks at 8 MB each). Storing all 12,500 chunk references directly in one filer entry would be unwieldy. Instead, SeaweedFS uses chunk manifests:

  1. The filer entry stores only the top-level chunks (say, one per 1 GB).
  2. Each top-level chunk has IsChunkManifest: true, meaning its blob content is a serialized list of sub-chunks.
  3. On read, NonOverlappingVisibleIntervals recursively resolves manifests to find the actual data chunks for the requested range.

This is transparent to the S3 API — it just sees a flat list of chunks after resolution.

Summary: Where Data and Metadata Diverge

PathFlows ThroughBottleneck?
Metadata (names, sizes, bucket config, IAM) S3 API → filer metadata store (Postgres/Redis/etc.) No — metadata store is horizontally scalable
Data (object bytes) Client → S3 API → volume servers (direct HTTP) No — volume servers are horizontally scalable, O(1) reads
File ID assignment S3 API → filer → master (one call per chunk) Lightweight — master only assigns IDs, doesn't touch data
