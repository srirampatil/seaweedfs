Lesson 5: Versioning Internals

S3 versioning preserves every version of every object. When you overwrite sunset.jpg, the old version isn't lost — it's kept in a .versions/ directory alongside the new one. This lesson traces how SeaweedFS implements this entirely on the filer, with no special volume-server support.

Versioning States

A bucket can be in one of three states:

StateExtended attribute valueBehavior
Never configured"" (absent)PUT overwrites; DELETE removes permanently
Enabled"Enabled"PUT creates new version; DELETE creates delete marker
Suspended"Suspended"PUT overwrites null version; DELETE creates null delete marker

Versioning is stored as an extended attribute on the bucket's filer directory entry. It's cached in BucketConfig.Versioning and read during every PutObject via getVersioningState().

Storage Layout

When versioning is enabled on bucket photos, the object sunset.jpg lives at:

/buckets/photos/sunset.jpg              ← empty (or null version if pre-versioning)
/buckets/photos/sunset.jpg.versions/    ← directory entry with latest-version pointer
/buckets/photos/sunset.jpg.versions/v_0x68a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5  ← version file
/buckets/photos/sunset.jpg.versions/v_0x68f1e2d3c4b5a6c7d8e9f0a1b2c3d4e5  ← older version

The .versions/ suffix is the constant s3_constants.VersionsFolder. Version files have the prefix v_ — the function getVersionFileName(versionId) returns "v_" + versionId.

The .versions/ directory itself is also a filer entry with extended attributes that act as the latest-version pointer:

// On the .versions/ directory entry:
ExtLatestVersionIdKey      = versionId          // which version is current
ExtLatestVersionFileNameKey = "v_0x68a1..."     // the filename of the current version
ExtLatestVersionSizeKey     = "1048576"          // cached size (avoids getEntry during list)
ExtLatestVersionMtimeKey    = "1721234567"       // cached mtime
ExtLatestVersionETagKey     = "\"abc123\""       // cached ETag
ExtLatestVersionOwnerKey    = "account123"       // cached owner
ExtLatestVersionIsDeleteMarker = "true"          // if current is a delete marker
Single-scan list efficiency: The cached size/mtime/ETag/owner on the .versions/ directory entry means ListObjects doesn't need to fetch each version file individually. For N versioned objects, this eliminates N extra getEntry calls — the metadata is directly usable from the directory listing.

Version ID Format

Version IDs are 32-hex-character strings. The first 16 chars encode a timestamp, the last 16 are random bytes for uniqueness:

versionId = invertedTimestamp(16 hex) + random(16 hex)

Two formats coexist:

FormatTimestamp encodingDetectionSort order
New (inverted)MaxInt64 - time.Now().UnixNano()first 16 chars > 0x4000...Smaller value = newer (lexicographic ascending = newest-first)
Old (raw)time.Now().UnixNano()first 16 chars < 0x4000...Smaller value = older (lexicographic ascending = oldest-first)
Why invert the timestamp? Filer directory listings are lexicographically sorted. By using MaxInt64 - nanos, newer versions naturally sort first in the directory listing without any custom sorting. The isNewFormatVersionId function detects the format by checking if the timestamp portion exceeds a threshold.

The getVersionIdFormat function checks the existing .versions/ directory to determine which format to use. New .versions/ directories always use inverted format. Existing directories inherit their format from the latest version ID already stored.

PutObject With Versioning Enabled

putVersionedObject (s3api_object_handlers_put.go:1457) does NOT write to the regular object path. Instead:

  1. Determine the version ID format (inverted or legacy).
  2. Generate a new version ID.
  3. Upload directly to .versions/v_<versionId> using putToFiler.
  4. The versionedFinalize callback runs after the version file is created:
    • Stamps the prior latest as noncurrent by writing ExtNoncurrentSinceNsKey on the demoted version entry — this gives the lifecycle engine an exact demotion timestamp.
    • Updates the .versions/ directory to point at the new version ID, file name, and cached metadata.

The regular object path (/buckets/photos/sunset.jpg) remains empty or carries only a pre-versioning null version. All versioned content lives under .versions/.

Delete With Versioning: Delete Markers

When versioning is enabled, deleting an object does NOT remove any data. Instead, it creates a delete marker — a zero-byte version file in .versions/ with ExtDeleteMarkerKey: "true":

createDeleteMarker(bucket, object):
  1. Generate new version ID
  2. Create v_<id> with ExtDeleteMarkerKey="true", zero chunks
  3. Update .versions/ pointer to this delete marker
  4. Set ExtLatestVersionIsDeleteMarker="true" on .versions/

On subsequent GET requests, getLatestObjectVersion returns the delete marker entry. The GET handler checks ExtDeleteMarkerKey and returns ErrNoSuchKey — exactly matching AWS behavior where the object appears deleted even though every historical version still exists.

To actually remove a version, the client must call DeleteObject?versionId=xxx. This permanently removes that specific version file from .versions/.

GetObject With Versioning

The read path branches based on versioning state (s3api_object_handlers.go:648):

Specific version requested (?versionId=xxx): getSpecificObjectVersion looks up .versions/v_xxx directly — O(1) lookup.

Latest version (no versionId): getLatestObjectVersion reads the .versions/ directory entry, extracts the pointer, and fetches the version file. It has a self-healing path with up to 8 retries if the pointer is stale (points to a file that no longer exists):

  1. Try to get the .versions/ directory entry (exponential backoff: 100ms, 200ms, 400ms...).
  2. Read ExtLatestVersionFileNameKey to find the current version file name.
  3. Fetch that version file.
  4. If the file doesn't exist (stale pointer): healStaleLatestVersionPointer rescans the .versions/ directory, finds the newest remaining entry, and updates the pointer. Returns a warning log entry so operators can track pointer drift.

Null Versions

The null version (versionId="null") represents objects that existed before versioning was enabled, or objects in a suspended-versioning bucket. These live at the regular object path, not under .versions/:

/buckets/photos/sunset.jpg   ← the null version (pre-versioning object)

When versioning is first enabled on a bucket, existing objects become null versions. New PUTs go to .versions/. The null version at the regular path persists until explicitly deleted.

Deleting Specific Versions

routedDeleteSpecificVersion (s3api_object_versioned_finalize.go:117) uses the repoint-before-delete pattern:

  1. Scan .versions/ to find the next-newest version (excluding the one being deleted).
  2. Repoint the pointer to the new latest BEFORE removing the blob. If the server crashes between steps 2 and 3, the pointer is already consistent — GETs serve the previous version or NoSuchKey correctly.
  3. Delete the version file from .versions/.
  4. If this was the last version (singleton): tear down the .versions/ directory.

For a multi-filer deployment, this happens as a single ObjectTransaction on the key's owner filer, serialized under the per-object lock.

DeleteMarker and Lifeycle

The ExtNoncurrentSinceNsKey timestamp lets the lifecycle engine compute NoncurrentDays precisely. When a new version or delete marker demotes the current latest, markVersionNoncurrent stamps the demoted entry with the exact nanosecond of demotion.
