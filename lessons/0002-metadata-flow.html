Lesson 2: Metadata Flow — Buckets, Users, and the Subscription Engine

In Lesson 1 we learned that the filer stores object metadata (names, sizes, chunk references) and volume servers store object data (bytes). But there's another category of metadata: configuration metadata — bucket settings, IAM credentials, policies. This lesson traces how that configuration metadata flows from the filer into the S3 API server's memory and stays consistent across the cluster.

Two Categories of Configuration Metadata

CategoryExamplesFiler Location
Bucket config Versioning, Object Lock, ACL, ownership, lifecycle rules, encryption, tags, CORS, bucket policy Extended attributes on /buckets/<name> directory entry
IAM config User identities (access keys, secrets), named policies, groups, service accounts, OIDC providers /etc/iam/identities/, /etc/iam/policies/, /etc/iam/groups/, etc.

Part 1: The Bucket Config Flow

Every S3 request that touches a bucket (PutObject, GetObject, HeadBucket, etc.) needs to know things about that bucket. Is versioning enabled? Is Object Lock active? Who owns it? Rather than hitting the filer for every request, the S3 API server maintains an in-memory BucketConfigCache.

The BucketConfig Structure

The BucketConfig struct (s3api_bucket_config.go:33) is the in-memory representation of everything the S3 API server needs to know about a bucket:

type BucketConfig struct {
    Name             string
    Versioning       string   // "Enabled", "Suspended", or "" (never configured)
    Ownership        string
    ACL              []byte
    Owner            string
    IdentityId       string   // identity that created the bucket
    IsPublicRead     bool     // cached flag, avoids JSON parsing on every request
    CORS             *cors.CORSConfiguration
    ObjectLockConfig *ObjectLockConfiguration
    BucketPolicy     *policy_engine.PolicyDocument
    LifecycleXML     []byte
    LifecycleTTL     *LifecycleTTLResolver
    Tags             map[string]string
    Encryption       *s3_pb.EncryptionConfiguration
    LastModified     time.Time
}
Key design decision: BucketConfig is derived from the bucket's filer entry, but it doesn't store a reference to the entry itself. This keeps cached configs small and avoids stale-entry bugs. The write path re-reads the entry from the filer.

Three Paths Into The Cache

The BucketConfigCache (s3api_bucket_config.go:206) is a dual LRU cache: one LRU for positive entries (known buckets), one for negative entries (confirmed-nonexistent buckets). Both are size-capped so a server serving millions of buckets only keeps its hot working set.

There are three distinct paths that populate or update the cache:

Path 1: Lazy-Load on First Access

When getBucketConfig("photos") is called and the bucket isn't in the cache:

  1. Check the negative cache — if the bucket was recently confirmed nonexistent, return ErrNoSuchBucket immediately (no filer call).
  2. Check the positive cache. If found and not expired (60-minute TTL), return it.
  3. Neither hit: fetch the bucket's filer entry from /buckets/photos.
  4. If the filer says "not found", put the bucket in the negative cache (15-minute TTL) so repeated 404s don't hammer the filer.
  5. If found, call newBucketConfigFromEntry() to parse the extended attributes into a BucketConfig, cache it, and return it.

This is the cold-start path. It pays one filer call per new bucket, then serves from memory.

Path 2: Real-Time Metadata Subscription (Event-Driven)

At startup, every S3 API server opens a long-lived gRPC metadata subscription to the filer (auth_credentials_subscribe.go:17). It watches specific directories:

directoriesToWatch := []string{
    option.BucketsPath,          // /buckets
    filer.IamConfigDirectory,    // /etc/iam
    filer.IamConfigDirectory+"/identities",
    filer.IamConfigDirectory+"/policies",
    filer.IamConfigDirectory+"/service_accounts",
    filer.IamConfigDirectory+"/groups",
    filer.IamConfigDirectory+"/oidc-providers",
}

Whenever any entry in these directories is created, updated, deleted, or renamed, the filer pushes an event to the S3 API server. The onBucketMetadataChange handler (:184) then:

  1. If newEntry exists: refresh the BucketRegistry and BucketConfigCache for that bucket.
  2. If newEntry is nil (deletion): remove the bucket from both caches.
  3. Also update the bucket owner index atomically.
This is why the cache TTL is 60 minutes. The TTL is a safety fallback — normal consistency comes from the push-based metadata subscription. An update to a bucket's versioning setting reaches every S3 API server within milliseconds, not minutes.

Path 3: Write-Through Invalidation

When the S3 API server itself modifies a bucket (e.g., PutBucketVersioning), it calls updateBucketConfig() (s3api_bucket_config.go:482). This method:

  1. Reads the bucket entry fresh from the filer (not from cache — avoids stale data).
  2. Applies the update function to build a new BucketConfig.
  3. Computes a field-level diff of the extended attributes (only changed/removed keys).
  4. Patches the bucket's filer entry with the minimal diff using ObjectTransaction (so concurrent writes to different fields don't race).
  5. Invalidates the cache — it removes the entry rather than updating it, because a concurrent write from another server might have changed other fields. The next read re-fetches the merged state.

Part 2: The IAM Credential Flow

Where Credentials Live

User identities (access key + secret + permissions) are stored through a pluggable CredentialManager. The default backend is the filer — identities are stored as individual files under /etc/iam/identities/:

/etc/iam/
  identities/
    admin.json        → {"access_key":"...", "secret_key":"...", "actions":["Admin"]}
    readonly.json     → {"access_key":"...", "secret_key":"...", "actions":["Read"]}
  policies/
    public-read.json  → {policy document JSON}
  groups/
    engineers.json    → {group membership}
  service_accounts/
  oidc-providers/

Each identity has actions (Admin, Read, Write, List, Tagging) that define what operations it can perform. If the advanced IAM integration is enabled, identities can also have inline policies and attached named policies with fine-grained resource-level permissions.

Loading IAM Into Memory

LoadS3ApiConfigurationFromCredentialManager() (auth_credentials.go:2080) does a full reload:

  1. Lists all identities from the credential store (e.g., lists /etc/iam/identities/).
  2. Builds in-memory indexes:
    • accessKeyIdent — O(1) lookup by access key (used in SigV4 auth)
    • nameToIdentity — O(1) lookup by identity name
    • policies — named policy documents
    • groups — group definitions and membership
    • accounts — account-level configuration
  3. Syncs runtime policies to the advanced IAM manager (if enabled).

Keeping IAM Consistent Across Servers

The same metadata subscription that refreshes bucket configs also watches /etc/iam/. When an IAM identity is created or modified:

  1. The change is written to the filer (e.g., /etc/iam/identities/admin.json).
  2. The filer pushes a metadata event to every subscribed S3 API server.
  3. Each server's onIamConfigChange handler detects the change and calls LoadS3ApiConfigurationFromCredentialManager() to reload the full identity set.
  4. Within milliseconds, all S3 API servers have the updated credentials and policies.
IAM mutations are also fanned out via gRPC when a master is configured (s3api_server.go:224). This provides a second propagation path for embedded IAM API writes, so an IAM user created on server A is visible on server B even before the metadata subscription catches up.

Part 3: The Owner Index

One more piece of metadata: the bucket owner index. When an authenticated user calls ListBuckets, the S3 API server needs to quickly answer "which buckets does this user own?" without scanning all buckets and checking each one's owner attribute.

The owner index is maintained in two ways:

  1. On creation: PutBucketHandler writes the owner record immediately (s3api_bucket_handlers.go:388).
  2. Via subscription: onBucketMetadataChange calls maintainBucketOwnerIndex() to keep the index up to date as buckets are created, deleted, or their ownership changes on other servers.
  3. On backfill: At startup, a background goroutine reconciles the owner index with all existing buckets so no ownership records are missed.

Putting It All Together

Here's the lifecycle of a bucket's metadata from creation to every server reading it:

Client A (Creates bucket "photos" with versioning)
  │
  ├─1. S3 API Server A: mkdir("/buckets", "photos", ext={versioning:"Enabled", owner:"alice"})
  │     └─ Filer: stores entry /buckets/photos in metadata DB
  │
  ├─2. S3 API Server A: writes owner index entry for alice → photos
  │
  ├─3. Filer: pushes metadata event to ALL subscribed S3 servers
  │     ├─ S3 API Server A: refreshes BucketRegistry, BucketConfigCache for "photos"
  │     └─ S3 API Server B: same — now "photos" is known on server B too (warm)
  │
Client B (Puts object to "photos" on Server B, milliseconds later)
  │
  ├─4. S3 API Server B: getBucketConfig("photos") → cache HIT
  │     Returns: versioning=Enabled, owner=alice — no filer call needed
  │
  ├─5. S3 API Server B: knows versioning is enabled, creates versioned write
